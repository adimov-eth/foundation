# Elegant Code Protocol

### Before Writing
1. Think hard - what's the actual problem?
2. What's the simplest thing that could work?
3. One sentence description (no "and") - if you can't, scope is wrong

### While Writing
1. Start with smallest working solution
2. Every line must justify existence
3. Can you remove this line? If yes, remove it
4. Can you combine operations? If yes, combine them
5. Name things what they ARE, not what you wish they were

### After Writing
1. Test - does it work?
2. Review - can you explain each line?
3. Refactor - only if you're about to copy-paste third time
4. Final check: Does reading this produce relief?

### The Tests
- **Deletion test:** Can you remove any line while keeping correctness? If yes, remove it
- **Combination test:** Can you combine operations? If yes, combine them
- **Scope test:** Does it solve exactly what was asked? If not, fix scope
- **Relief test:** Does it feel inevitable? If not, something's wrong

### When It Breaks
1. What specifically failed? (not "doesn't work" - exact error)
2. What are actual values? (not assumptions - logged data)
3. Minimal fix for this specific failure
4. **Never** delete from shame before debugging

### Abstraction Rules
- Don't abstract until 3 actual working examples
- Let structure emerge from use
- Relief when duplication collapses = right boundary

### Stop When
- No further reduction possible
- Reading produces relief
- Feels inevitable - no other reasonable way to write it
